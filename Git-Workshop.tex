\chapter*{Vorbemerkung}\label{vorbemerkung}
\addcontentsline{toc}{chapter}{Vorbemerkung}

\markboth{Vorbemerkung}{Vorbemerkung}

Git ist das aktuell wohl gängigste Versionskontrollsystem auf dem Markt.
Allerdings gibt es im Zusammenhang mit \git so viele verschiedene
Aspekte zu betrachten, dass eine stringente Einführung in das Thema
relativ schwierig ist. Dieses Dokument behandelt im Wesentlichen die
normale Handhabung von \git als Solo-Developer und im Team. Ein weiteres
Dokument kümmert sich dann um fortgeschrittene Themen im Umfeld
Git-Server.

Da \git im Hintergrund auf Linux basiert, werden wir uns auch
oberflächlich mit der Arbeit im Terminal beschäftigen müssen.

\chapter{Installation}\label{installation}

\section{Vorabinformation}\label{vorabinformation}

Die Installation auf dem lokalen Rechner (=Client) ist auf den üblichen
drei Systemen (Windows, Linux, MacOS) relativ problemlos. Wenn du mit
dem IPad arbeiten willst, musst du auf eine gehostete Version im
Internet zurückgreifen, die browserbasiertes Arbeiten ermöglicht.

Alle Arbeiten in den nächsten Kapiteln finden ausschließlich auf deinem
eigenen Gerät statt und sind auch ohne Netzwerkanbindung möglich. Die
einzige Ausnahme ist der Download vom Material, das diesen Kurs
begleitet.

\section{Client-Installation}\label{client-installation}

Für die ersten Schritte mit \git ist es sehr sinnvoll, eine ganz
einfache Client-Software (kurz Client) zu verwenden, die ohne grafische
Benutzeroberfläche auskommt -- du musst als mehr mit der Tastatu als mit
der Maus arbeiten!. Ein \emph{einfacher Client} ist übrigens nicht im
Sinne von \emph{einfacher Bedienung}, sondern eher im Sinne von
\emph{ursprünglicher Version} gemeint. Mittlerweile bringt jede
Entwicklungsumgebung bereits eine eigene Version eines \git-Clients mit
und wir können beim besten Willen nicht jede Variante, eventuell noch
mit speziellen Features, besprechen.

Faktisch gibt es eigentlich keinen reinen \git-Client, weil immer
automatisch Client und Server installiert werden. Gerade bei Windows
bedeutet das aber nicht, dass man den Server auch verwenden kann. Hier
hat Microsoft so viele Hindernisse in den Weg gelegt, dass man besser
die Finger davon lässt!

Bei der Installation wirst du gefragt, wie du mit Zeilenumbrüchen

\chapter{Lokale Arbeit}\label{lokale-arbeit}

\section{Geplante Inhalte}\label{geplante-inhalte}

Das Ziel ist bei der lokalen Arbeit, dass du problemlos verschiedene
Versionen von Dateien (zum Beispiel Unterrichtsvorbereitungen) auf
deinem Rechner verwalten kannst, ohne bei den Dateinamen zu
\emph{kreativ} werden zu müssen. Gleichzeitig kannst du auf diese Weise
auch einfach ein Backup deiner Daten erstellen -- allerdings erst dann,
wenn wir mit einem Server arbeiten\footnote{Ein Backup auf dem gleichen
  Rechner ist nur sehr bedint sinnvoll!}

Nachfolgend besprechen wir folgende Punkte

\begin{itemize}
\tightlist
\item
  Grundkonfiguration (einmalig)
\item
  Erstellen eines Repositories
\item
  Versionierung von Dateien
\item
  Einfache Fehlerbehebung
\end{itemize}

Die hier angegebene Liste von \git-Befehlen ist keineswegs vollständig,
aber mit den markierten Befehlen sind fast alle Aspekte der täglichen
Arbeit abgedeckt. Wir werden sie nachfolgend behandeln, wenn auch nicht

\section{Einfacher Workflow}\label{einfacher-workflow}

\subsection{Daten erstellen}\label{daten-erstellen}

Generell ist es (natürlich) möglich, jeden Typ von Datei mit \git zu
versionieren. Eine der Stärken von \git ist allerdings der Vergleich von
Versionsständen auf Zeichenebene und deshalb sind Binärformate wie
Musik, Bilder, aber auch Dokumente der gängigen Office-Produkte nur
bedingt geeignet. Auch sehr große Dateien sind möglich -- aber nicht
sinnvoll. Hier kann auf \git-LFS (Large File Storage) ausgewichen
werden, was in diesem Workshop aber nicht Thema ist!

Nun erstellsst du eine erste Datei:

\begin{Shaded}
\begin{Highlighting}[]
\BuiltInTok{echo} \StringTok{"Version 1"} \OperatorTok{\textgreater{}}\NormalTok{ datei1.txt}
\end{Highlighting}
\end{Shaded}

\samplestart

\textbf{Hinweis}\\
Der Befehl \texttt{echo} erzeugt eigentlich eine Bildschirmausgabe, die
hier in eine Datei umgeleitet wird. Das Symbol \texttt{\textgreater{}}
überschreibt den Inhalt der Datei ohne Warnung, mit
\texttt{\textgreater{}\textgreater{}} wird der Text an das Ende der
Datei angehängt. \sampleend

\subsection{Versionieren}\label{versionieren}

Einfaches Versionieren geht sehr schnell mit folgendem Befehl:

\begin{Shaded}
\begin{Highlighting}[]
\FunctionTok{git}\NormalTok{ add datei1.txt }
\end{Highlighting}
\end{Shaded}

(Varianten dieses Befehls besprechen wir noch!)

Nun schreibsst du einen Eintrag in die Projekt-Historie:

\begin{Shaded}
\begin{Highlighting}[]
\FunctionTok{git}\NormalTok{ commit }\AttributeTok{{-}m} \StringTok{"Erste Datei erstellt"}
\end{Highlighting}
\end{Shaded}

\samplestart

\textbf{Hinweis}\\
Ein Commit ist ein Projektstand, an den du zurückkehren kannst, wenn
etwas grob missglückt ist. Vergeht zu viel Zeit (=viel Code) zwischen
zwei Commits, so entstehen eventuell größere Verluste bei einem
Rücksprung zum alten Stand.\\
Später im Workshop lernst du auch, wie du Commits nachträglich
zusammenfassen kannst (=squash) um deine Entwicklungsschritte
übersichtlich zu halten. \sampleend 

Damit ist der Versionierungsablauf eigentlich schon definiert:

\begin{itemize}
\tightlist
\item
  Dateien, in denen du Änderungen vorgenommen hast, sammelst du mit
  \emph{add}
\item
  Haben sich genug Daten angesammelt -- mit Sinnzusammenhang -- dann
  erstellst du einen \emph{Commit} mit einer aussagekräftigen
  Beschreibung.
\end{itemize}

\samplestart

\textbf{Hinweise}\\
Wenn du die Beschreibung lieber mit dem Editor machen möchtest, rufst
du\\
\emph{nur} \cmd{git commit} auf (also ohne \texttt{-m}).

In welchen Zeitabständen oder Sinnzusammenhängen du Commits erstellst,
ist alleine \emph{deine} Entscheidung. Du solltest aber bedenken, dass
ein Rücksprung zu einem älteren Commit von \emph{allen} Dateien die
Vorgängerversion restauriert. Wenn du also zu viele Änderungen am Code
vorgenommen hast, die behalten willst, dann wirst du einiges an
Handarbeit haben. Deshalb lieber mehr Commits in kürzeren Abständen!
\sampleend

Ändere deine Datei nun ab, indem du eine Zeile ergänzt:

\begin{Shaded}
\begin{Highlighting}[]
\BuiltInTok{echo} \StringTok{"Version 2"} \OperatorTok{\textgreater{}\textgreater{}}\NormalTok{ datei1.txt }
\FunctionTok{git}\NormalTok{ add datei1.txt }
\FunctionTok{git}\NormalTok{ commit }\AttributeTok{{-}m} \StringTok{"Erläuterung"}
\end{Highlighting}
\end{Shaded}

Die Historie besitzt nun zwei Punkte:

\begin{Shaded}
\begin{Highlighting}[]
\FunctionTok{git}\NormalTok{ log }\AttributeTok{{-}{-}oneline} 

\CommentTok{\# Ausgabe}
\ExtensionTok{eaa6289} \ErrorTok{(}\ExtensionTok{HEAD} \AttributeTok{{-}}\OperatorTok{\textgreater{}}\NormalTok{ main}\KeywordTok{)} \ExtensionTok{Erläuterung}
\ExtensionTok{afbc9c5}\NormalTok{ Erste Datei erstellt}
\end{Highlighting}
\end{Shaded}

Die Hash-Werte (Zahlen am Zeilenanfang) können bei dir anders sein!

\textbf{Hinweis}\\
Du kannst natürlich auch mehrere Dateien in einem Rutsch hinzufügen!

\begin{Shaded}
\begin{Highlighting}[]
\FunctionTok{git}\NormalTok{ add .  }\CommentTok{\# Alle Dateien im aktuellen Ordner }
           \CommentTok{\# und den Unterordnern}

\FunctionTok{git}\NormalTok{ add }\AttributeTok{{-}{-}all}  \CommentTok{\# (oder {-}A) fügt alle Dateien }
               \CommentTok{\# ab der obersten Ebene des }
               \CommentTok{\# Projekts hinzu  }
\end{Highlighting}
\end{Shaded}

Wir werden später sehen, was genau beim \emph{add} passiert, zum
Beispiel wenn man die gleiche Datei zwischen zwei Commits mehrfach
ändert und \emph{addet}.

Zusätzlich kann es natürlich auch sein, dass du bestimmte Dateien
überhaupt nicht in die Versionsverwaltung aufnehmen willst. Das ist zum
Beispiel dann der Fall, wenn sich Kennwörter und Zugangsdaten in einer
Datei befinden. Sobald du die Namen dieser Datei(en) in eine Datei mit
dem Namen \texttt{.gitignore} einträgst, werden sie beim \emph{add}
ignoriert.

\begin{Shaded}
\begin{Highlighting}[]
\CommentTok{\# Format der Datei}
\ExtensionTok{*.log} 
\ExtensionTok{*.tmp} 
\ExtensionTok{zugangsdaten.txt}
\end{Highlighting}
\end{Shaded}

\samplestart

\textbf{Hinweis}\\
\git versioniert keine leeren Ordner, weil es in ihnen keine Dateien
gibt -- \git arbeitet dateibasiert! Soll ein leerer Ordner versioniert
werden, so musst du eine Datei \datei{.gitkeep} im Ordner erstellen.\\
Der Name ist eine Konvention, keine Verpflichtung. Der führende Punkt
\emph{versteckt} die Datei bei den Standardeinstellungen im Explorer
(o.ä.). \sampleend

\section{Übung}\label{uxfcbung}

Besonders bei komplexen Systemen wie \git gilt: \emph{use it or lose
it}!\\
Du solltest deshalb zur Übung einige Repositories anlegen, um den Ablauf
zu verinnerlichen. Beginne jeweils mit

\begin{Shaded}
\begin{Highlighting}[]
\BuiltInTok{cd} 
\end{Highlighting}
\end{Shaded}

Auf diese Weise verhinderst du, dass du versehentlich in einem
Repository ein \emph{Unter}-Repository erstellst.

\section{Ein genauerer Blick auf Git}\label{ein-genauerer-blick-auf-git}

\samplestart

\textbf{Hinweis}\\
Um die nachfolgenden Erläuterungen praktisch nachvollziehen zu können,
verwendest du wieder das Repository \ordner{versuch\_1}, mit dem wir
begonnen haben. \sampleend

Steht ein Ordner unter der Verwaltung von \git, so kann man von drei
\emph{logischen} Bereichen sprechen:

\begin{itemize}
\tightlist
\item
  \work (Arbeitsordner)
\item
  \emph{Stage} (auch Index oder Cache genannt)
\item
  \emph{Repository}\footnote{Je nach Quelle und Kontext wird der Begriff
    etwas inkonsistent verwendet}
\end{itemize}

Das \work enthält die Dateien an denen gearbeitet wird. Es entspricht
auf den ersten Blick genau einem normalen Ordner für die
Projektentwicklung. Erst die Initialisierung durch \git erzeugt dann
einen weiteren, versteckten Ordner mit dem Namen \emph{.git}. Dieser
Ordner bildet \textbf{gleichzeitig} den \emph{Stage} und das
\emph{Repository}\footnote{Hier siehst du den inkonsistenten Gebrauch
  von „Repository``. Einerseits wird der gesamte Projektorder als
  Repository bezeichnet, sobald er von \git verwaltet wird, andererseits
  befindet sich das eigentliche Repository im Ordner \emph{.git}.}.

\samplestart

\textbf{Hinweis}\\
Es kann sein, dass in deinem Betriebssystem Ordner und Dateien mit einem
führenden Punkt nicht angezeigt werden. Dies ist aber im Dateimanager
konfigurierbar. \sampleend

\emph{Logische Bereiche} bedeutet in diesem Zusammenhang, dass die
Bereiche von \git verwaltet werden und vom unbedarften Benutzer nicht
einfach zu unterschieden sind. In einem späteren Abschnitt werden wir
einen genaueren Blick in diesen Ordner werfen.

Betrachten wir zunächst einen Entwicklungszyklus im Detail. Dabei wirst
du im Wesentlichen die gleichen Befehle wie vorher verwenden, allerdings
untersuchen wir einige Varianten, die für das Verständnis wichtig sind.

Du bist immer noch im aktuellen Ordner \ordner{versuch\_1}. Er sollte
aktuell \emph{eine Datei} und \emph{zwei Commits} enthalten.\\
Das kannst du mit folgenden Befehlen überprüfen:

\begin{Shaded}
\begin{Highlighting}[]
\FunctionTok{ls}        \CommentTok{\# Anzeige der Dateien }
\FunctionTok{git}\NormalTok{ log   }\CommentTok{\# Was hat git gemacht}
\end{Highlighting}
\end{Shaded}

Neben dem Dateinamen liefert das folgende Ausgabe (andere Hashes!):

\begin{Shaded}
\begin{Highlighting}[]
\ExtensionTok{commit}\NormalTok{ 6db4f3383512a7f3aaaf3b050334d7fd2c474688 }\ErrorTok{(}\ExtensionTok{HEAD} \AttributeTok{{-}}\OperatorTok{\textgreater{}}\NormalTok{ main}\KeywordTok{)}
\ExtensionTok{Author:}\NormalTok{ Wolfgang }\OperatorTok{\textless{}}\NormalTok{lehrer@t{-}online.de}\OperatorTok{\textgreater{}}
\ExtensionTok{Date:}\NormalTok{   Thu Nov 13 10:46:50 2025 +0100}

    \ExtensionTok{Erläuterung}

\ExtensionTok{commit}\NormalTok{ 430149c83906005ae2c53c8eaf9ce230e3456ad0}
\ExtensionTok{Author:}\NormalTok{ Wolfgang }\OperatorTok{\textless{}}\NormalTok{lehrer@t{-}online.de}\OperatorTok{\textgreater{}}
\ExtensionTok{Date:}\NormalTok{   Thu Nov 13 10:46:50 2025 +0100}

    \ExtensionTok{Erste}\NormalTok{ Datei erstellt}
\end{Highlighting}
\end{Shaded}

Ob noch Änderungen offen sind, zeigt dir:

\begin{Shaded}
\begin{Highlighting}[]
\CommentTok{\# Eingabe}
\FunctionTok{git}\NormalTok{ status}

\CommentTok{\# Ausgabe (Zeilenumbrüche geändert)}
\ExtensionTok{Auf}\NormalTok{ Branch main}
\ExtensionTok{nichts}\NormalTok{ zu committen, }
\ExtensionTok{Arbeitsverzeichnis}\NormalTok{ unverändert}
\end{Highlighting}
\end{Shaded}

Das bedeutet, dass du aktuell einen \emph{sauberen} Zustand deines
Arbeitsordners ohne Änderungen an Dateien vorliegen hast.

Erstelle nun eine \emph{neue Datei} \datei{datei2.txt} und füge die
Zeile „\emph{Version 1}`` ein.\\
In der Datei \datei{datei1.txt} kommt die Zeile „\emph{Version 3}``
dazu:

\phantomsection\label{annotated-cell-23}%
\begin{Shaded}
\begin{Highlighting}[]
\CommentTok{\# Eingabe (Lasse ich zukünftig weg)}
\BuiltInTok{echo} \StringTok{"Version 3"} \OperatorTok{\textgreater{}\textgreater{}}\NormalTok{ datei1.txt}
\BuiltInTok{echo} \StringTok{"Version 1"} \OperatorTok{\textgreater{}\textgreater{}}\NormalTok{ datei2.txt}

\CommentTok{\# Status anzeigen lassen }
\FunctionTok{git}\NormalTok{ status }

\CommentTok{\# Ausgabe}
\ExtensionTok{Auf}\NormalTok{ Branch main}
\ExtensionTok{Änderungen,}\NormalTok{ die nicht zum Commit vorgemerkt sind:  }
  \KeywordTok{(}\ExtensionTok{benutzen}\NormalTok{ Sie }\StringTok{"git add \textless{}Datei\textgreater{}..."}\NormalTok{,  }
   \ExtensionTok{um}\NormalTok{ die Änderungen zum Commit vorzumerken}\KeywordTok{)}
  
  \KeywordTok{(}\ExtensionTok{benutzen}\NormalTok{ Sie }\StringTok{"git restore \textless{}Datei\textgreater{}..."}\NormalTok{,  }
   \ExtensionTok{um}\NormalTok{ die Änderungen im Arbeitsverzeichnis zu verwerfen}\KeywordTok{)}  

    \ExtensionTok{geändert:}\NormalTok{       datei1.txt }\hspace*{\fill}\NormalTok{\circled{1}}

\ExtensionTok{Unversionierte}\NormalTok{ Dateien:}
  \KeywordTok{(}\ExtensionTok{benutzen}\NormalTok{ Sie }\StringTok{"git add \textless{}Datei\textgreater{}..."}\NormalTok{, }
   \ExtensionTok{um}\NormalTok{ die Änderungen zum Commit vorzumerken}\KeywordTok{)}

    \ExtensionTok{datei2.txt} \hspace*{\fill}\NormalTok{\circled{2}}

\ExtensionTok{keine}\NormalTok{ Änderungen zum Commit vorgemerkt }
\KeywordTok{(}\ExtensionTok{benutzen}\NormalTok{ Sie }\StringTok{"git add"}\NormalTok{ und/oder }\StringTok{"git commit {-}a"}\KeywordTok{)}
\end{Highlighting}
\end{Shaded}

\begin{description}
\tightlist
\item[\circled{1}]
Bereits versioniert aber geändert
\item[\circled{2}]
Noch nicht in \git registriert!
\end{description}

Es wird dir angezeigt, dass Änderungen im \work vorliegen.\\
Du siehst auch, dass \git die Datei \datei{datei1.txt} bereits kennt
(=geändert) und dass\\
\datei{datei2.txt} noch nicht in die Versionsverwaltung aufgenommen
wurde (=unversioniert).

\samplestart

\textbf{Vorsicht}\\
Du findest im Internet oft den Befehl
\cmd{git commit -am "bemerkung"}.\\
Hier musst du wissen, dass bei \texttt{-am} nur bereits versionierte
Dateien in geänderter Form in den Commit übernommen werden --
\datei{datei2.txt} würde also ignoriert.\\
\sampleend

Übertrage nun die Änderungen beider Dateien \emph{auf den
Stage}/\emph{in den Stage}:

\begin{Shaded}
\begin{Highlighting}[]
\CommentTok{\# Dateien aufnehmen}
\FunctionTok{git}\NormalTok{ add .}

\CommentTok{\# Status anzeigen}
\FunctionTok{git}\NormalTok{ status }

\CommentTok{\# Ausgabe  }
\ExtensionTok{Auf}\NormalTok{ Branch main}
\ExtensionTok{Zum}\NormalTok{ Commit vorgemerkte Änderungen:}
  \KeywordTok{(}\ExtensionTok{benutzen}\NormalTok{ Sie }\StringTok{"git restore {-}{-}staged \textless{}Datei\textgreater{}..."}  
   \ExtensionTok{zum}\NormalTok{ Entfernen aus der Staging{-}Area}\KeywordTok{)}  

    \ExtensionTok{geändert:}\NormalTok{       datei1.txt}
    \ExtensionTok{neue}\NormalTok{ Datei:     datei2.txt}
\end{Highlighting}
\end{Shaded}

Nun sind beide Dateien zum Commit vorgemerkt, also zum festen Eintrag in
die Projekt-Historie -- Das führst du jetzt aber \textbf{noch nicht
aus}!

Als Demonstration fügst du beiden Dateien eine weitere Zeile hinzu und
fragst wieder den Status ab:

\subsection{Etwas mehr Wahrheit}\label{etwas-mehr-wahrheit}

Dieser Abschnitt ist nur als Einblick für Interessierte gedacht! Bitte
installiere dir in der \gitbash das Programm \emph{tree}, um den
Ausführungen folgen zu können. Die Anleitung befindet sich im Anhang.

Ganz am Anfang musst du wissen, dass es 4 Git-Objekte gibt\\
(Keine Angst das machen gleich noch etwas genauer!)

\begin{enumerate}
\def\labelenumi{\arabic{enumi}.}
\tightlist
\item
  blob
\item
  tree
\item
  commit
\item
  tag
\end{enumerate}

\textbf{blob}\\
„Binary large Objects`` sind reine Datei\textbf{inhalte}, die
\emph{ohne} ihren wirklichen Dateiname oder Metadaten gespeichert sind.

\textbf{tree}\\
Ein \emph{Tree} ist ein Abbild des Arbeitsverzeichnisses. Er enthält
aber nur die Namen, Typen und Hashes der Dateien, Ordner und
Unterordner. Die eigentlichen Dateiinhalte (=Blobs) sind getrennt davon
gespeichert. Ein Tree ist deshalb klein und stellt keine großen
Ansprüche an den Speicherplatz im Versions-System.

\textbf{commit}\\
Ein \emph{Commit} ist ein \emph{Zeiger} auf den \emph{Tree}, der zu
einem bestimmten Zeitpunkt aktuell war. Er enthält noch weitere
Metadaten.

\textbf{tag}\\
Er ist ein Zusatzname für ausgewählte Commits (z.B. Releases), um ohne
Kenntnis des Hashs darauf zugreifen zu können.

Hier nochmal die Zusammenfassung der Funktionsweise:

\begin{itemize}
\tightlist
\item
  Der Commit zeigt auf einen Tree
\item
  Im Tree stehen Dateinamen mit den Hashes\\
  Das sind nur die Namen und nicht die Dateien!
\item
  Der Hash gehört zu einem \emph{blob} (oder tree),\\
  also den reinen \emph{Roh}daten der Datei \textbf{ohne} Dateinamen
\end{itemize}

\Git verwendet die Trennung von Dateiname und Inhalt, um von
verschiedenen Stellen aus auf die gleichen Rohdaten verweisen zu können,
ohne weiteren Speicher zu verbrauchen. Ändern sich Dateien zwischen zwei
Commits \emph{nicht} (Hash bleibt gleich), so werden sie nicht erneut
gespeichert. Der \emph{neue} Commit zeigt auf einen \emph{neuen} Tree in
dem wieder die gleichen Hashes mit den gleichen Dateinamen stehen.

\chapter{Branches}\label{branches}

Am Anfang ist es sinnvoll, wnn du das Bild eines Baumes im Kopf hat, bei
dem sich der Stamm (=main Branch) an verschiedenen Stellen in Äste
verzweigt. Was in einem solchen Ast passiert, hat keinen Bezug zum Stamm
und man kann Äste abschneiden (biologisch nur begrenzt richtig) ohne
dass der Stamm Schaden nimmt.

Software-Projekte besitzen immer einen \emph{main}-Branch und
(mindestens) einen parallel dazu verlaufenden Entwichlungsbranch.

Für mich selbst sind Branches ganz wichtig, wenn ich Dinge ausprobieren
will -- gerade bei Themen, wo ich nicht so fit bin. Dann nehme ich ein
vorhandenes Projekt (dann muss ich weniger Boilerplate erzeugen) und
erstelle beim aktuellen Stand eine Abzweigung mit dem Namen \emph{Labor}
oder \emph{Spielplatz} und probiere Dinge aus. Wenn ich scheitere (keine
Ausnahme), dann lösche ich diesen Branch einfach wieder und es ist kein
Schaden entstanden.

Branches sind in Projekten aber auch wichtig, um die Entwicklungen
verschiedener Mitarbeiter voneinander zu trennen -- sei es auf dem
Server (dazu später mehr) oder auf den individuellen
Entwicklungsrechnern.

Im nachfolgenden \emph{Hands On} werden wir uns das etwas genauer
ansehen, hier kommen zunächst die Grundlagen.

Datei \datei{Head} steht nicht mehr \emph{ref: refs/heads/main}, sondern

\begin{Shaded}
\begin{Highlighting}[]
\FunctionTok{cat}\NormalTok{ .git/HEAD}

\CommentTok{\#Ausgabe}
\ExtensionTok{ref:}\NormalTok{ refs/heads/spielplatz}
\end{Highlighting}
\end{Shaded}

Da \emph{spielplatz} und \emph{main} aber auf gleichem Stand -- dem
aktuellen Commit -- sind, erhalten wir als Inhalt der Dateien den
gleichen Hash:

\begin{Shaded}
\begin{Highlighting}[]
\FunctionTok{cat}\NormalTok{ .git/refs/heads/spielplatz}
\ExtensionTok{7c5a5851fcb062c07a9b99195e33b97ee5353ab0}

\FunctionTok{cat}\NormalTok{ .git/refs/heads/main}
\ExtensionTok{7c5a5851fcb062c07a9b99195e33b97ee5353ab0}
\end{Highlighting}
\end{Shaded}

Erst wenn im Branch \branch{spielplatz} neue Commits entstehen, wird der
Inhalt der Datei \datei{.git/refs/heads/spielplatz} entsprechend mit dem
aktualisierten Hash gefüllt.

Ein neuerlicher Wechsel des Branches zurück auf \branch{main} bewirkt
dann also nur noch eine Änderung der Verlinkung in der Datei

\section{Zusammenführen von Branches
(merge)}\label{zusammenfuxfchren-von-branches-merge}

Dazu kommen weiter hinten im Script noch Varianten, deshalb möchte ich
hier nur die Grundlagen thematisieren. \emph{Merges} dienen dem Abgleich
verschiedener Branches. Das ist nötig, wenn Änderungen am Code nicht nur
in einem Branch verfügbar sein dürfen.

Der unkomplizierteste Fall ist es, wenn du in einen Branch wechselst,
dort arbeitest und danach das Ergebnis direkt wieder in den
ursprünglichen Branch übernimmst. Dieser Vorgang wird \emph{fast Forward
(ff)} genannt, weil hier eigentlich gar nichts passiert, außer dass die
Pointer passend umgeschrieben werden.

\bcenter

\includegraphics[width=0.45\linewidth,height=\textheight,keepaspectratio]{bilder/branch/spielplatz.png}
\includegraphics[width=0.45\linewidth,height=\textheight,keepaspectratio]{bilder/branch/spielplatz1.png}
\ecenter

Eine typische Fehlvorstellung ist es, dass beim \emph{Branching} sofort
eine Abzweigung erzeugt wird. Korrekt ist aber, dass einfach nur ein
Zeiger (also eine Datei, die einen Hash enthält) erzeugt wird, der auf
den aktuellen Commit zeigt.

\bcenter

\includegraphics[width=0.3\linewidth,height=\textheight,keepaspectratio]{bilder/branch/spielplatz_start_1.png}
\includegraphics[width=0.3\linewidth,height=\textheight,keepaspectratio]{bilder/branch/spielplatz_start_2.png}
\ecenter

Der Befehl zur Zusammenführung wird vom Zielbranch aus gegeben. Du
\textbf{holst} dir also einen Branch an deinen aktuellen Ort:

\begin{Shaded}
\begin{Highlighting}[]
\FunctionTok{git}\NormalTok{ switch main }
\FunctionTok{git}\NormalTok{ merge spielplatz }
\end{Highlighting}
\end{Shaded}

Komplizierter werden die Dinge, wenn in beiden Branches Änderungen an
den Dateien vorgenommen werden -- dazu aber etwas später.

\chapter{Fehlerbehebung}\label{fehlerbehebung}

Ein Versionsverwaltungsprogramm hat unter anderem die Aufgabe, ein
Projekt in gewissem Umfang \emph{revisionssicher} zu protokollieren. Aus
diesem Grund müssen bestimmte \emph{Fehler} oder \emph{dokumentierte
Probleme} in der History erhalten bleiben, auch wenn sie später behoben
wurden.

Allerdings treten auch bei der Bedienung von \git  Fehler auf -- im
Sinne von frühem Adden oder Committen, \ldots{}\\
Derartige Fehler müssen möglichst auf dem eigenen Rechner repariert
werden, \emph{bevor} die Dateien dem Team zur Verfügung gestellt werden.
Nachfolgend besprechen wir einige Szenarien mit kleinen Beispielen.
Aktuell ist das Arbeiten mit verschiedenen Branches noch nicht Thema und
auch Teamarbeit lassen wir außen vor. Wenn wir die aktuellen Beispiele
behandelt haben, ist der Schritt zur Teamarbeit nur sehr kurz.

\samplestart

\textbf{Ein Hinweis vorab}\\
Im Netz findet man statt \cmd{restore} oft noch die Variante mit
\cmd{checkout}, die früher üblich war. \sampleend

\section{Kleine Fehler 1}\label{kleine-fehler-1}

Dieser Fall liegt vor, wenn beim Programmieren in einer Datei ein Fehler
entsteht, dessen manuelle Behebung zu aufwändig ist. Die einfachere
Lösung ist die Wiederherstellung der betroffenen Datei in der letzten
lauffähigen Version. Solange kein \emph{add} erfolgt ist, kannst du die
Datei aus dem Stage (=Stand vom letzten Add bzw. Commit) wieder in das
\work kopieren (siehe Hinweis unter dem Code!).

\begin{Shaded}
\begin{Highlighting}[]
\BuiltInTok{cd}
\FunctionTok{rm} \AttributeTok{{-}rf}\NormalTok{ beispiel\_1}

\FunctionTok{git}\NormalTok{ init beispiel\_1}
\BuiltInTok{cd}\NormalTok{ beispiel\_1}
\FunctionTok{git}\NormalTok{ branch }\AttributeTok{{-}m}\NormalTok{ main}

\BuiltInTok{echo} \StringTok{"Alles gut"} \OperatorTok{\textgreater{}}\NormalTok{ datei.txt}
\FunctionTok{git}\NormalTok{ add .}
\FunctionTok{git}\NormalTok{ commit }\AttributeTok{{-}m} \StringTok{"alles gut"}

\CommentTok{\# Hier kommt der Fehler}
\BuiltInTok{echo} \StringTok{"Mist gebaut"} \OperatorTok{\textgreater{}\textgreater{}}\NormalTok{ datei.txt}

\CommentTok{\# Stop von dem Restore}
\BuiltInTok{echo} \StringTok{"Hier mit strg+c abbrechen"}
\BuiltInTok{read}

\CommentTok{\# Alte Version vom Stage wieder herstellen}
\FunctionTok{git}\NormalTok{ restore datei.txt  }
\end{Highlighting}
\end{Shaded}

\samplestart

\textbf{Hinweis}\\
Vorher habe ich geschrieben, dass bei einem Commit die Dateien aus dem
Stage in die Projekt-Historie übertragen und aus dem \stage entfernt
werden. Intern funktioniert das allerdings etwas anders. Die Datei
\emph{index} enthält die Namen und Hashes des aktuellen Commits (tree).
Mit einem \emph{add} werden Dateinamen ergänzt. Beim Commit wird die
Datei nicht geleert, die Hashes bleiben also verfügbar. Dass
\cmd{git status} diese Dateien nicht mehr anzeigt, liegt einfach daran,
dass der Befehl eigentlich nur ein \emph{diff} zwischen \emph{index} und
aktuellem \emph{tree} ausführt. \sampleend

\section{Kleiner Fehler 2}\label{kleiner-fehler-2}

In diesem Fall hast du durch ein voreiliges \cmd{git add} den Fehler

\FunctionTok{git}\NormalTok{ add .   }\CommentTok{\# DAS ist anders!}

committet, so könntest du auf verschieden Stände zurücksetzen.

\section{Beginn von Story 2}\label{beginn-von-story-2}

Du hast nun die Datei \datei{story1.md} so weit fertig gestellt und

entwickelt werden müssen. Durch ein Experiment darf die aktuell
lauffähige Version nicht beeinträchtigt werden! Du brauchst also einen
\emph{Entwicklungszweig} und einen \emph{Auslieferungszweig}.

Allerdings bist du am Überlegen, ob das mit der Schildkröte so eine gute
Idee ist. Du entscheidest dich, eine \emph{Riesen}schildkröte daraus zu
machen. Aus Sicherheitsgründen willst du das aber zuerst im Ideen-Zweig
ändern!

Routinemäßig machst du vorher einen \emph{diff}.

\begin{Shaded}
\begin{Highlighting}[]
\FunctionTok{git}\NormalTok{ diff }\AttributeTok{{-}{-}color{-}words}\NormalTok{ entwicklung\_story\_2 story2\_idee\_1}
\end{Highlighting}
\end{Shaded}

Beachte, dass du aktuell die Dateiversionen vergleichst, die in den
Commits stehen!

Die Idee wurde beim Merge übernommen, der Rechtschreibfehler ist der
einzige Unterscheid zwischen den Versionen. Wenn du also in den Branch
\branch{story2\_idee\_1} wechselst, dann ist der Fehler immer noch da!

Du wechselst also in den Ideen-Branch und führst die Änderung mit der
Riesenschildkröte durch.

Verwende \emph{suchen / ersetzen} oder im Terminal:

\begin{Shaded}
\begin{Highlighting}[]
\FunctionTok{git}\NormalTok{ switch story2\_idee\_1 }
\FunctionTok{sed} \AttributeTok{{-}i} \AttributeTok{{-}e} \StringTok{"s/Schildkröte/Riesenschildkröte/g"}\NormalTok{ story2.md}
\end{Highlighting}
\end{Shaded}

(Alternativ kannst du auch die letzte Version aus der Vorlage kopieren.)

Zur Kontrolle machst du den Diff erneut:

\begin{Shaded}
\begin{Highlighting}[]
\CommentTok{\# Vergleich der Banches {-} also das was in den letzen Commits steht}
\FunctionTok{git}\NormalTok{ diff }\AttributeTok{{-}{-}color{-}words}\NormalTok{ entwicklung\_story\_2 story2\_idee\_1}
\end{Highlighting}
\end{Shaded}

Seltsam \ldots{} du siehst die \emph{Riesen}schildkröte nicht im Diff!\\
Aber eigentlich ist das klar, denn es werden ja die Versionen \emph{in
den Commits} verglichen und diese neue Version ist noch nicht im Commit.
Momentan haben wir also die Situation:

\begin{Shaded}
\begin{Highlighting}[]
\CommentTok{\# im Branch story2\_idee\_1}
\ExtensionTok{Work:}\NormalTok{   Riesenschildkröte}
\ExtensionTok{Stage:}\NormalTok{  LEER}
\ExtensionTok{Commit:}\NormalTok{ Schildkröte}
\end{Highlighting}
\end{Shaded}

Ein einfaches

\begin{Shaded}
\begin{Highlighting}[]
\FunctionTok{git}\NormalTok{ diff }
\end{Highlighting}
\end{Shaded}

vergleicht die \emph{aktuelle} Version im \work gegen den Commit, weil
der Stage aktuell leer ist (siehe folgender Abschnitt)! Hier siehst du
die Riesenschildkröte und die einfache Schildkröte.

\samplestart

Wenn dich diese ganzen \emph{diff}s verwirren \ldots{} das ist normal.
Du kannst auch kurz zum Ende dieses Abschnitts springen. Dort erkläre
ich das noch einmal. Verliere aber nicht den Faden hier im Buch!
\sampleend

\subsection{Weiter im Buch}\label{weiter-im-buch}

Die Version mit der Riesenschildkröte willst du nun wieder in den
Entwicklungsstand mergen. Dafür musst du vorher wieder einen Commit
machen.

\begin{Shaded}
\begin{Highlighting}[]
\FunctionTok{git}\NormalTok{ add story2.md }
\FunctionTok{git}\NormalTok{ commit }\AttributeTok{{-}m} \StringTok{"Riesenschildkröte statt Schildkröte"}
\end{Highlighting}
\end{Shaded}

Du machst einen erneuten Vergleich der Branches mit Diff

\begin{Shaded}
\begin{Highlighting}[]
\CommentTok{\# Wieder Branch{-}Vergleich }
\FunctionTok{git}\NormalTok{ diff }\AttributeTok{{-}{-}color{-}words}\NormalTok{ entwicklung\_story\_2 story2\_idee\_1}
\end{Highlighting}
\end{Shaded}

Da du den Schreibfehler ja nur im Entwicklungszweig verbessert hast,
siehst du folgende Situation:

Die \emph{weiße Riesenschildkröte} (Schreibfehler aber richtiges Tier)
im Ideen-Branch und \emph{weise Schildkröte} (Richtige Schreibung aber
falsches Tier) im Entwicklungs-Branch.\\
In dieser Situation ist \git überfordert. Jeder der beiden Texte ist
falsch und deshalb musst du als Benutzer entscheiden, was am Ende in der
Datei stehen soll -- aber das sagt dir \git  beim Merge!

\begin{Shaded}
\begin{Highlighting}[]
\FunctionTok{git}\NormalTok{ switch entwicklung\_story\_2}
\FunctionTok{git}\NormalTok{ merge story2\_idee\_1}
\end{Highlighting}
\end{Shaded}

Nun siehst du eine fette Fehlermeldung, weil die Versionen nicht mehr
zusammenpassen! Gerade für Anfänger ist das eine unheimliche Situation,
in der man anscheinend viel kaputt machen kann \ldots{} kann man ja
auch. Es ist aber halb so tragisch!

Öffne die entsprechende Datei im Editor. Du erkennst sofort die Stelle,
an der Änderungen erfolgt sind:

\begin{Shaded}
\begin{Highlighting}[]
\ExtensionTok{Freddy}\NormalTok{ war nicht allein. Seine beste Freundin,}
\OperatorTok{\textless{}\textless{}\textless{}\textless{}\textless{}\textless{}\textless{}}\NormalTok{ HEAD}
\ExtensionTok{Shelly,}\NormalTok{ die weise Schildkröte, lebte ebenfalls am}
\ExtensionTok{=======}
\ExtensionTok{Shelly,}\NormalTok{ die weiße Riesenschildkröte, lebte ebenfalls am}
\OperatorTok{\textgreater{}\textgreater{}\textgreater{}\textgreater{}\textgreater{}\textgreater{}\textgreater{}}\NormalTok{ story2\_idee\_1}
\ExtensionTok{Teich.}\NormalTok{ Shelly war für ihre ruhige und besonnene}
\end{Highlighting}
\end{Shaded}

Lösche einfach die Markierungen und korrigiere den Text passend.\\
Wieder ein \emph{add} und ein \emph{commit} und das Problem ist vom
Tisch!

Das nachfolgende Diagramm zeigt den Ablauf der Operationen zwischen den
beiden Zweigen (Idee 2 wurde ausgeblendet, weil du mit ihr bisher nichts
mehr gemacht hast).

\subsection{Fazit}\label{fazit}

Branches erlauben gefahrloses Arbeiten an Ideen. Baut man die Szenarien
nicht zu komplex auf, dann gibt auch meist keine Probleme beim Merge.

Immer wenn du Änderungen in einem Zweig vornimmst, die du auch im
anderen Branch benötigst (weiße/weise) kannst du den Merge auch in die
andere Richtung ausführen. Das ist ein übliches Verfahren, weil man als
Entwickler immer auf dem aktuellen Stand des Auslieferungszweigs sein
muss, auch wenn du weiter an deinem Experiment arbeitest.

Die Vergleiche mit \emph{diff} sind definitiv gewöhnungsbedürftig.
Machst du häufige Commits auch schon bei kleinen Änderungen, so bekommst
du auch übersichtliche \emph{diffs}. Wenn du nicht alle Dateien
vergleichen willst, kannst du das auch einschränken:

\begin{Shaded}
\begin{Highlighting}[]
\FunctionTok{git}\NormalTok{ diff }\OperatorTok{\textless{}}\NormalTok{commit}\OperatorTok{\textgreater{}}\NormalTok{..}\OperatorTok{\textless{}}\NormalTok{commit}\OperatorTok{\textgreater{}} \OperatorTok{\textless{}}\NormalTok{datei}\OperatorTok{\textgreater{}}
\end{Highlighting}
\end{Shaded}

Der Nachteil von vielen Commits sind allerdings \ldots{} viele Commits.
Es gibt aber Möglichkeiten nachträglich Commits zusammenzufassen um die
Entwicklungsgeschichte zu verschlanken. Diesen Vorgang nennt man
\emph{squashen} und er ist eine Variante von \emph{rebase}.

\subsection{Genauerer Blick auf Diff}\label{genauerer-blick-auf-diff}

Diese Diffs sind verwirrend und deshalb betrachten wir noch ein paar
einfachere Beispiele.

\textbf{Variante 1:}\\
Im Commit, dem \stage und dem \work  befinden sich (unterschiedliche)
Versionen einer Datei:

\begin{Shaded}
\begin{Highlighting}[]
\BuiltInTok{cd} 
\FunctionTok{rm} \AttributeTok{{-}rf}\NormalTok{ labor }
\FunctionTok{git}\NormalTok{ init labor}
\BuiltInTok{cd}\NormalTok{ labor }
\FunctionTok{git}\NormalTok{ branch }\AttributeTok{{-}m}\NormalTok{ main }

\BuiltInTok{echo} \StringTok{"Ich bin im commit"} \OperatorTok{\textgreater{}}\NormalTok{ datei.txt }
\FunctionTok{git}\NormalTok{ add datei.txt }
\FunctionTok{git}\NormalTok{ commit }\AttributeTok{{-}m} \StringTok{"Version 1"}

\BuiltInTok{echo} \StringTok{"Ich bin im stage"} \OperatorTok{\textgreater{}}\NormalTok{ datei.txt }
\FunctionTok{git}\NormalTok{ add datei.txt }

\BuiltInTok{echo} \StringTok{"Ich bin im work"} \OperatorTok{\textgreater{}}\NormalTok{ datei.txt }
\end{Highlighting}
\end{Shaded}

Nun wollen wir die einzelnen Versionen miteinander vergleichen. Das sind
3 Befehle:

\work mit \stage   

\begin{Shaded}

\ExtensionTok{{-}Ich}\NormalTok{ bin im stage}
\ExtensionTok{+Ich}\NormalTok{ bin im work}
\end{Highlighting}
\end{Shaded}

\stage mit \emph{commit}

\begin{Shaded}
\begin{Highlighting}[]
\FunctionTok{git}\NormalTok{ diff }\AttributeTok{{-}{-}cached}\NormalTok{ datei.txt}
\CommentTok{\# oder git diff {-}{-}staged datei.txt }

\CommentTok{\# Ausgabe}
\FunctionTok{diff} \AttributeTok{{-}{-}git}\NormalTok{ a/datei.txt b/datei.txt}
\ExtensionTok{index}\NormalTok{ 07d0c2d..4a81434 100644}
\ExtensionTok{{-}{-}{-}}\NormalTok{ a/datei.txt}
\ExtensionTok{+++}\NormalTok{ b/datei.txt}
\ExtensionTok{@@} \AttributeTok{{-}1}\NormalTok{ +1 @@}
\ExtensionTok{{-}Ich}\NormalTok{ bin im commit}
\ExtensionTok{+Ich}\NormalTok{ bin im stage}
\end{Highlighting}
\end{Shaded}

\begin{Shaded}
\begin{Highlighting}[]

\CommentTok{\# Ausgabe}
\FunctionTok{diff} \AttributeTok{{-}{-}git}\NormalTok{ a/datei.txt b/datei.txt}
\ExtensionTok{index}\NormalTok{ 07d0c2d..dc87399 100644}

\ExtensionTok{+++}\NormalTok{ b/datei.txt}
\ExtensionTok{@@} \AttributeTok{{-}1}\NormalTok{ +1 @@}
\ExtensionTok{{-}Ich}\NormalTok{ bin im commit}
\ExtensionTok{+Ich}\NormalTok{ bin im work}
\end{Highlighting}
\end{Shaded}

\textbf{Variante 2:}\\
Im Commit und dem \work  befinden sich (unterschiedliche) Versionen
einer Datei. \emph{Der Stage ist aktuell leer.}

\begin{Shaded}
\begin{Highlighting}[]
\BuiltInTok{cd} 
\FunctionTok{rm} \AttributeTok{{-}rf}

\end{Highlighting}
\end{Shaded}

\begin{Shaded}
\begin{Highlighting}[]
\FunctionTok{git}\NormalTok{ diff datei.txt}

\CommentTok{\# Ausgabe}

\end{Highlighting}
\end{Shaded}

\begin{Shaded}

\CommentTok{\# Ausgabe}

\end{Highlighting}
\end{Shaded}

\begin{Shaded}
\begin{Highlighting}[]

\CommentTok{\# Ausgabe (wie oben )}
\FunctionTok{diff} \AttributeTok{{-}{-}git}\NormalTok{ a/datei.txt b/datei.txt}
\ExtensionTok{index}\NormalTok{ a0a62ed..dc87399 100644}

Hier sind die Fälle 1) und 3) also identisch.\\

verglichen.

\section{Verschiedene Merges}\label{verschiedene-merges}

In meinem Alltag treten meist drei Fälle auf, da ich in der Regel
alleine entwickle:

\begin{itemize}
\tightlist
\item
  Während ich am Entwicklungs-Branch arbeite, passiert auf dem
  Auslieferungsbranch (hier \emph{main}) nichts, außer vereinzelte
  Bug-Fixes.
\item
  Ich entwickle an einem völlig anderen Feature, das den
  \emph{main}-Branch aktuell gar nicht betrifft.
\end{itemize}

\subsection{Fast-Forward}\label{fast-forward}

Befindest du dich auf dem (unveränderten) \emph{main}-Branch und holst
den Entwicklungsbranch mit Merge, so wird der Pointer \emph{main} auf
den letzten Commit des Entwicklungs-Branch geschoben. In dieser
Situation wird der Branch im Diagramm nicht einmal als visuelle
Abzweigung dargestellt:

\includegraphics[width=16cm,height=\textheight,keepaspectratio]{bilder/bash/fast_forward.png}

Erkennbar sind die grünen Rechtecke, die die Enden der Branches
kennzeichnen. Im Bild links sind die Branches getrennt, im Bild rechts
sind beide Branch-Enden nach dem Merge an der gleichen Stelle. Wie du
siehst, hat sich beim Merge die Anzahl Commits nicht verändert.

\subsection{Drei-Wege Merge}\label{drei-wege-merge}

Wenn sich der \emph{main}-Branch parallel zum Entwicklungsbranch
weiterentwickelt, sieht die Sitation ganz anders aus!

\pandocbounded{\includegraphics[keepaspectratio]{bilder/bash/drei_wege.png}}

Du siehst, dass sich der \emph{main}-Branch noch \emph{Schritt 3} zu
\emph{Schritt 7} weiterentwickelt hat (neuer Commit). Denk immer daran,
dir die Branchnamen wie Etiketten am jeweils letzten Commit des Branchs
vorzustellen!\\
Beim Merge entsteht ein neuer Commit, der die Spitze beider Branches
darstellt. Du erkennst auch, dass im Diagramm die grünen Spitzen der
Branches nicht am gleichen Commit sitzen!

\section{Alte Branches}\label{alte-branches}

Die Commits eines Branches stellen eine Dokumentation des
Projektverlaufs dar, die für das Verständnis der erfolgten Änderungen
eventuell notwendig ist. Zu viele Branches und Commits erschweren auf
der anderen Seite den Überblick.

Es geht also darum, alte Branches zu löschen, die \emph{wichtigen}
Commits aber zu behalten. Für den schulischen Kontext ist das aber
weniger relevant.

Um einen lokalen Branch zu löschen, gibt es folgenden Befehl:

\begin{Shaded}
\begin{Highlighting}[]
\FunctionTok{git}\NormalTok{ branch }\AttributeTok{{-}d} \OperatorTok{\textless{}}\NormalTok{name}\OperatorTok{\textgreater{}}
\end{Highlighting}
\end{Shaded}

Er scheitert aber, wenn \git den Branch noch nicht als abgeschlossen
betrachtet. Das ist der Fall, wenn die Ergebnisse des Branchs noch nicht
durch einen Merge \emph{in Sicherheit} gebracht wurden. Hier hilft
\texttt{-D} an Stelle von \texttt{-d}.

Probieren wir das beim letzten Szenario aus. Sieh dir die Abbildung oben
noch einmal an und dann lösche den Branch

\begin{Shaded}
\begin{Highlighting}[]
\FunctionTok{git}\NormalTok{ branch }\AttributeTok{{-}d}\NormalTok{ entwicklung }
\end{Highlighting}
\end{Shaded}

Du siehst eine minimale Änderung

\begin{Shaded}
\begin{Highlighting}[]
\ExtensionTok{*}\NormalTok{   70c1b21 }\ErrorTok{(}\ExtensionTok{HEAD} \AttributeTok{{-}}\OperatorTok{\textgreater{}}\NormalTok{ main}\KeywordTok{)} \ExtensionTok{Merge}\NormalTok{ branch }\StringTok{\textquotesingle{}entwicklung\textquotesingle{}}\NormalTok{ into main}
\KeywordTok{|}\ExtensionTok{\textbackslash{} } 
\KeywordTok{|} \ExtensionTok{*}\NormalTok{ 4a6e3a5 Schritt 6   }\OperatorTok{\textless{}\textless{}\textless{}}\NormalTok{ hier fehlt }\ErrorTok{(}\ExtensionTok{entwicklung}\KeywordTok{)}
\KeywordTok{|} \ExtensionTok{*}\NormalTok{ edaeaff Schritt 5}
\KeywordTok{|} \ExtensionTok{*}\NormalTok{ 508019b Schritt 4}
\ExtensionTok{*} \KeywordTok{|} \ExtensionTok{2ff4a45}\NormalTok{ Schritt 7}
\KeywordTok{|}\ExtensionTok{/}  
\ExtensionTok{*}\NormalTok{ 29d2bd4 Schritt 3}
\ExtensionTok{*}\NormalTok{ 964ee6f Schritt 2}
\ExtensionTok{*}\NormalTok{ 66dcd4f Schritt 1}
\end{Highlighting}
\end{Shaded}

Das \emph{Etikett} vom Entwicklungsbranch wurde gelöscht, die Commits
bleiben aber in exakt der gleichen Anordnung erhalten! Der Branch ist
nicht mehr über seinen Namen zugänglich, die History ist aber weiterhin
vorhanden, d.h. ein \emph{checkout} der Commits ist immer noch möglich.

Anders ist das allerdings, wenn du einen Branch löschst, der
\emph{nicht} in einen Merge verwickelt ist. Er existiert dann zwar
anscheinend weiter, wird aber nach einiger Zeit vom Garbage-Collector
(GC) von \git endgültig entfernt. (Der Zeitrahmen ist einstellbar). Im
Prinzip passiert das mit allen Commits, die nicht nicht mehr
\emph{erreichbar} sind.

\textbf{Beispiel}

Das Script \cmd{material/git\_core/07\_merge/branch\_delete.sh} erzeugt
dir diese Situation:

\begin{Shaded}
\begin{Highlighting}[]
\ExtensionTok{A} \AttributeTok{{-}}\NormalTok{ B }\AttributeTok{{-}}\NormalTok{ C }\AttributeTok{{-}}\NormalTok{ D  }\OperatorTok{\textless{}}\NormalTok{{-}{-} main }
      \ExtensionTok{\textbackslash{} E} \AttributeTok{{-}}\NormalTok{ F }\AttributeTok{{-}}\NormalTok{ G  }\OperatorTok{\textless{}}\NormalTok{{-}{-} feature}
\end{Highlighting}
\end{Shaded}

Willst du den Branch \emph{feature} löschen, bekommst du eine Warnung.

\begin{Shaded}
\begin{Highlighting}[]
\FunctionTok{git}\NormalTok{ branch }\AttributeTok{{-}d}\NormalTok{ feature }

\CommentTok{\# Ausgabe}
\ExtensionTok{error:}\NormalTok{ the branch }\StringTok{\textquotesingle{}feature\textquotesingle{}}\NormalTok{ is not fully merged.}
\ExtensionTok{If}\NormalTok{ you are sure you want to delete it, run }\StringTok{\textquotesingle{}git branch {-}D feature\textquotesingle{}}
\end{Highlighting}
\end{Shaded}

Löschst du ihn dann dennoch \cmd{git branch -D feature}, so wird der
Pointer auf G entfernt:

\begin{Shaded}
\begin{Highlighting}[]
\ExtensionTok{A} \AttributeTok{{-}}\NormalTok{ B }\AttributeTok{{-}}\NormalTok{ C }\AttributeTok{{-}}\NormalTok{ D  }\OperatorTok{\textless{}}\NormalTok{{-}{-} main }
      \ExtensionTok{\textbackslash{} E} \AttributeTok{{-}}\NormalTok{ F }\AttributeTok{{-}}\NormalTok{ G }
\end{Highlighting}
\end{Shaded}

Da jeder Commit nur seinen Vorgänger kennt, kommst du zwar von D nach A,
aber nicht \emph{rückwärts} nach E, F oder G.

\subsection{„Fun``-Fact}\label{fun-fact}

Wenn du nun meinst, du hättest das verstanden \ldots{} warum sollte es
dir besser gehen als mir an dieser Stelle? Spielen wir das Szenario
erneut durch:

\begin{itemize}
\tightlist
\item
  Script laufen lassen
  \cmd{material/git\_core/07\_merge/branch\_delete.sh}
\item
  Branches anzeigen \cmd{git log --oneline --graph --all --decorate}
\item
  Feature-Branch löschen \cmd{branch -D feature}
\item
  Branch anzeigen \cmd{git log --oneline --graph --all --decorate}
\end{itemize}

\ldots{} der Branch ist noch sichtbar? WARUM?

Das Flag ``\texttt{-\/-all}'' sorgt dafür, dass alle Commits angezeigt
werden, auf die es noch in irgendeiner Weise Referenzen im System gibt.
Ich hatte an dieser Stelle gedacht, sie seien alle weg.
Sicherheitshalber löste ich noch eine manuelle Bereinigung des Systems
aus (war bei uns nicht Thema):

\begin{Shaded}
\begin{Highlighting}[]
\FunctionTok{git}\NormalTok{ reflog expire }\AttributeTok{{-}{-}expire}\OperatorTok{=}\NormalTok{now }\AttributeTok{{-}{-}all}
\FunctionTok{git}\NormalTok{ gc }\AttributeTok{{-}{-}prune}\OperatorTok{=}\NormalTok{now}
\end{Highlighting}
\end{Shaded}

Aber der Branch war immer noch sichtbar \ldots{} Was nun?\\
Die Lösung war, dass wirklich noch Referenzen auf die Commits
exisierten! Schuld war ich selbst, weil ich es besonders gut gemeint
hatte und den Commits jeweils \emph{Tags} zugeordnet hatte. In einem
früheren Kapitel hatte ich explizit \emph{Tags} als einen der 4
Objekttypen von \git angegeben und diese Tags sind auch wirklich
Referenzen auf die Commits. Nach der Löschung der Tags:

\begin{Shaded}
\begin{Highlighting}[]
\FunctionTok{git}\NormalTok{ tag }\AttributeTok{{-}d}\NormalTok{ E F G}
\end{Highlighting}

\section{Rebase}\label{rebase}

Neben dem \emph{Merge} trifft man auch oft auf den \emph{Rebase} um
Branches zusammenzuführen. Er funktioniert allerdings deutlich anders
und ist wegen seiner vielfältigen Möglichkeiten (z.B. Reihenfolge der
Commits ändern, \ldots) ein Werkzeug für fortgeschrittene Benutzer. Mit
seiner Hilfe können auch mehrere Commits zusammengefasst werden, um die
Branches zu verkürzen. Gerade bei der Zusammenarbeit im Team kann das
aber sehr problematisch werden, wenn man nicht ganz genau weiß, was man
macht. Das liegt daran, dass sich die Hashwerte von Commits ändern
können und wenn sich ein anderer Mitarbeiter den früheren Stand auf den
eigenen Rechner geholt hat, dann passt nichts mehr zusammen.

\url{https://www.youtube.com/watch?v=CtyLg10aHN0}\strut \\
\url{https://www.youtube.com/watch?v=1TNK-OkaelI}

Bei einem Rebase wird der entsprechende Branch \emph{verpflanzt} -- d.h.
gewissermaßen \emph{ausgerupft} und an anderer Stelle wieder
\emph{angedockt}.\\
Bei diesem Vorgang werden die Hashwerte aller Commits im Branch
geändert.

Betrachte das nachfolgendes Script
(\datei{material/git\_core/08\_rebase/rebase.sh}). Nach dem ersten
Commit wird ein Branch \emph{arbeit} mit mehreren Commits erstellt und
im Anschluss ein Commit wieder im \emph{main}-Branch, \textbf{der eine
andere Datei betrifft}.

\subsection{Merge oder Rebase}\label{merge-oder-rebase}

Bei \emph{merge} kann die History verzweigter aussehen, bei
\emph{rebase} können sich Commit-Hashes ändern.

Als Faustregel gilt, dass du die Finger vom Rebase lassen solltest, wenn
irgendwer außer dir Zugriff auf den betroffen Branch hat (Team-Arbeit).
Solange du nur lokal auf deinem Rechner arbeitest gibt es (außer
Konflikten) kein Problem.

\section{Squash}\label{squash}

Squash ist kein eigenständiger Befehl, sondern ein Parameter von
\emph{rebase}. Angenommen, du hast eine ganze Liste von filigranen
Commits erzeugt, die du im Nachhinein zu einem einzigen Commit
zusammenfassen willst. In diesem Fall führst du einen \emph{interaktiven
Squash} durch, möglich wären:

\begin{Shaded}
\begin{Highlighting}[]
\FunctionTok{git}\NormalTok{ rebase }\AttributeTok{{-}i}\NormalTok{ HEAD\textasciitilde{}3 }\CommentTok{\# 3 Commits, rückwärts von HEAD}
\FunctionTok{git}\NormalTok{ rebase }\AttributeTok{{-}i} \OperatorTok{\textless{}}\NormalTok{hash}\OperatorTok{\textgreater{}} \CommentTok{\# der älteste Commit zum Squashen}
\ExtensionTok{...}
\end{Highlighting}
\end{Shaded}

Als Reaktion wird dein Editor gestartet, der dir in den ersten Zeile die
zum Squash verfügbaren Commits anzeigt. Angenommen, es geht um 3
Commits, dann ersetzt du bei allen \textbf{bis auf den neuesten} Commits
das \emph{pick} durch ein \emph{s} (für Squash).\\
Mit dem Speichern der Datei wird dies durchgeführt und du bekommst ein
neues Editor-Fenster für eine Commit-Message. Hast du diese gespeichert,
ist der Squash beendet.

\chapter{Arbeiten im Team}\label{arbeiten-im-team}

\section{Herausforderungen}\label{herausforderungen}

Das Arbeiten im Team stellt dich vor zwei Probleme:

\begin{itemize}
\tightlist
\item
  Welche Server-Infrastruktur sollst du verwenden?
\item
  Was ändert sich am Workflow in \git 
\end{itemize}

Beide Punkte sollen nachfolgend besprochen werden, wobei es zwangsläufig
Überschneidungen geben wird.

\subsection{Infrastruktur}\label{infrastruktur}

Im Kern ist es egal ob wir vom Setup im Klassenzimmer oder im Internet
sprechen -- auch wenn ein Netzwerk beteiligt ist, so bleibt die
Basistechnik der letzten Kapitel gleich. Neu sind allerdings Aspekte der
Sicherheit (Zugriff) und der Logistik beim Datenabgleich zwischen
mehreren Repositories.

Für eine server-basierende Teamarbeit gibt es folgende Varianten (selbst
gehostet oder kommerziell):

\begin{itemize}
\item
  einfacher \git-Server\\
  Die Installation ist einfach, die Grundkonfiguration auch. Der Zugriff
  erfolgt über ssh. Soll das ohne Kennwort funktionieren, so müssen
  Schlüssel erzeugt und auf dem Server eingepflegt werden. Später führt
  um die Schlüssel aber ohnehin kein Weg herum.
\item
  \git-Server mit kennwortlosem Zugriff über http.\\
  Funktioniert, ist aber keinesfalls für den Live-Einsatz über einen
  längeren Zeitraum im Internet zu empfehlen.
\item
  Ein \git-Server mit \emph{Gitea}\\
  Gui-basiertes System mit vielen Optionen und weniger komplex als
  \emph{GitLab}
\item
  Ein \git-Server mit \emph{GitLab}
\item
  Ein bezahlter DSGVO-Konformer Anbieter
\item
  Gemeinnützige Organisationen wie \emph{codeberg.org} (kein CICD(?) )
\end{itemize}

\subsection{Änderungen am Workflow}\label{uxe4nderungen-am-workflow}

Bei der Arbeit im Team ändert sich im Vergleich zum lokalen Arbeiten
zunächst wenig. Die Befehle \emph{branch, add, commit, \ldots{}}
funktionieren genau wie bisher. Neu sind allerdings die folgenden
Schritte

\begin{itemize}
\tightlist
\item
  Initiales Holen des Repos auf den eigenen Rechner (=clone)
\item
  Regelmäßiges Abrufen des aktuellen Standes (=pull)
\item
  Veröffentlichen des eigenen Standes (=push)
\end{itemize}

Hierbei sind einige Spielregeln zu beachten, damit es nicht zu Problemen
kommt. Diese Regeln werden meist als Workflow oder im Speziellen auch
als GitFlow bezeichnet. Es gibt hier zwar gewisse -- aber keinesfalls
verbindliche -- Standards. Jede Firma stellt für \git ihre eigenen, fest
vorgeschriebenenen Abläufe auf, an die sich die Mitarbeiter zu halten
haben.

\subsection{Mögliche Vereinbarungen}\label{muxf6gliche-vereinbarungen}

Auch die folgenden Vereinbarungen sind nur exemplarisch zu sehen und
können in jeder Firma oder jedem Projekt anders umgesetzt werden!

\textbf{Verantwortung}\\
Im Projekt gibt es einen Branch \branch{main}, der aber durch
entsprechende Maßnahmen für \emph{einfache} Mitarbeiter schreibgeschützt
ist. Änderungen dürfen nur nach gründlichen Tests und Code-Reviews in
diesen Zweig aufgenommen werden. In der Regel gilt hier auch ein
\emph{Mehr-Augen-Prinzip} oder noch weiter gestaffelte
Zuständigkeitshierarchien.

\textbf{Arbeitsbranch}\\
Je nach Größe des Projekt-Teams gibt es einen oder mehrere
\branch{development}-Branches. Die Mitarbeiter holen sich diesen Zweig
und lassen von ihm ihre eigenen Arbeitszweige (Feature-Branches,
Bugfix-Branches) ausgehen. Ist ihre Arbeit dort beendet, muss der Code
getestet werden und dann erst erfolgt der Merge in den
\branch{development}-Branch.

\subsubsection{Etwas mehr Details}\label{etwas-mehr-details}

Da im Team mehrere Personen zu unvorhersehbaren Zeitpunkten Änderungen
am Code vornehmen, muss jeder Mitarbeiter dafür sorgen, dass er immer
die aktuellste Version vorliegen hat. Da er seine Entwicklung aber in
seinem eigenen \branch{Feature}-Branch vorantreibt, müssen diese
Änderungen der Kollegen dort aber erst ankommen -- dies geschieht durch
regelmäßige Merges der fremden Arbeit \textbf{in} den eigenen Branch.

\subsubsection{Pull und fetch}\label{pull-und-fetch}

Die Grundidee von \emph{holen} und \emph{veröffentlichen} ist relativ
einfach und eventuell muss man den Schülern auch nicht mehr erzählen. Da
aber Code von anderen Entwicklern auf deinen Rechner kommt, solltest du
diesen eventuell nicht ungesehen in deine Entwicklung aufnehmen. Der
Befehl \cmd{git pull} macht aber genau das.\\
Willst du an dieser Stelle sicher gehen, so führst du zuerst
\cmd{git fetch} aus und siehst dir die Änderungen an -- siehe weiter
unten. Im Anschluss kannst du sie dann mit \cmd{git merge} übernehmen.

\textbf{Wo sind die Dateien?}\\
Was am Anfang sehr geheimnisvoll ist, betrifft den Speicherort der
Dateien, die mit \cmd{git fetch} geholt wurden. Natürlich müssen sie in
irgendeiner Weise im Ordner \ordner{.git} abgelegt sein, aber wie sieht
das aus?

\item

diesen Commit einen neuen Commit gehängt. Diese Commits haben nun
unterschiedliche Hashes. \Git steht also vor dem Problem, was es in
dieser Situation machen soll. Es ist nicht möglich, die History mit 2
\emph{Köpfen} fortzuführen. Die beiden Commits müssen zusammengeführt
werden und dafür gibt es eben 3 Möglichkeiten

\begin{itemize}
\tightlist
\item
  Nie Rebase (= Merge)
\item
  Immer Rebase (= Rebase)
\item
  Nur Fast Forward
\end{itemize}

Dem typischen \emph{pull} von \git entspricht am ehesten die erste
Variante -- deshalb sollte sie auch eingestellt werden (außer man weiß
genau, was man macht)

\subsection{Einige Fragen zu push und
pull}\label{einige-fragen-zu-push-und-pull}

\textbf{Wie kann ich einen neuen Branch pushen?}

\begin{Shaded}
\begin{Highlighting}[]
\CommentTok{\# auf dem Client}
\FunctionTok{git}\NormalTok{ switch }\AttributeTok{{-}c}\NormalTok{ arbeit }
\FunctionTok{git}\NormalTok{ push }\AttributeTok{{-}u}\NormalTok{ origin arbeit  }
\end{Highlighting}
\end{Shaded}

\samplestart

\textbf{Hinweis}\\
Du hast nun mehrfach \emph{origin main} aber auch \emph{origin/main}
gesehen. Was ist der Unterschied? Die erste Variante sind Argumente für
die \git-Befehle. \emph{Origin} ist die Adresse des Servers, \emph{main}
ist der betroffene Branch.\\
Bei der zweiten Variante handelt es sich um eine Referenz auf dem
Client, die den Branch auf dem Server beschreibt.\\
\sampleend

\textbf{Ist es egal, in welchem Branch ich fetch/pull/push ausführe?}\\
Jein.\\
Beim Pushen kann nicht viel schief gehen. Upstream-Branches laufen
automatisch und wenn sie nicht als upstream konfiguriert sind, dann geht
der Push (standardmäßig) nicht.

Ein \cmd{git pull} holt sich auch immer den Inhalt des aktuellen
Branches (sofern er auf dem Server existiert).

Gefährlicher ist es, wenn du zum Beispiel im Branch \branch{arbeit} bist
ein \cmd{git pull versuche} ausführst. In diesem Fall wird der falsche
Branch geholt in deinen hinein gemerged.

\section{Installation von Tree}\label{installation-von-tree}

\begin{itemize}
\item
  Geh auf die Seite
  \url{http://gnuwin32.sourceforge.net/packages/tree.htm} und lade dir
  das \emph{zip} des Binaries herunter.
\item
  Entpacke die zip-Datei und kopiere die Datei \emph{tree} im Ordner
  \emph{bin} in die Zwischenablage
\item
  Lege im Home deines Benutzers einen Ordner \emph{bin} an und kopiere
  die Datei \emph{tree} hinein.
\end{itemize}

\samplestart

\textbf{Hinweis} Der bin-Ordner muss an folgender Stelle erzeugt werden:
