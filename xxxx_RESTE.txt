### Kurze Wiederholung zum Projektaufbau

In (klassischem) Java werden Quelldateien mit der Endung *java* 
erstellt und mit dem Befehl *javac* in *Class*-Dateien übersetzt, 
die dann mai *java* gestartet werden können. Das Erstellen von 
Jar-Archiven ist hier erst einmal nicht Thema.

Für TDD sollten wir -- besonders im Hinblick auf den späteren 
Einsatz des Build-Systems *Maven* bereits die passende 
Ordnerstruktur auch für einfache Projekte verwenden.

Durch das Script \datei{Java_ordner.sh} kann eine entsprechende
Ordnerstruktur für ein Projekt mit dem Namen \ordner{java\_01}
erstellt werden.

Der eigentliche Source-Code liegt dann im Ordner

```bash
java_01/src/main/java/de/dillingen/alp/kurs
```

und die späteren Tests analog unter

```bash
java_01/src/test/java/de/dillingen/alp/kurs
```

Für die Tests werden die *jar*-Dateien im Ordner \ordner{src/lib}
abgelegt.

Das Compilieren der Quelldateien erfolgt mit folgendem Befehl:

``bash
javac -d out $(find src/main/java -name "*.java")
javac -d out -cp "out:lib/*" $(find src/test/java -name "*.java")
```

\samplestart
**Hinweis**  
Bei Windows müssen die Befehle anders lauten, das *find* 
Befehl dort in dieser Form nich existiert.
Gerade bei sehr wenigen Dateien kann es eine Alternative sein,
die Dateien dediziert per Hand anzugeben. Statt dem Doppelpunkt 
muss übrigens ein Semicolon verwendete werden!
\sampleend

```bash
javac -d out src/main/java/de/dillingen/alp/kurs/Main.java
javac -d out -cp "out;lib/*" src/test/java/de/dillingen/alp/kurs/MainTest.java
java -cp "out;lib/*" org.junit.platform.console.ConsoleLauncher --scan-class-path
```

Wir starten mit der einfachst möglichen Konfiguration, bestehend aus 

* StringFlipper.java = Java-Klasse
* StringFlipperTest.java = Testdatei
* junit5.jar

Die drei Dateien befinden sich in einem Ordner und können für den 
Workshop aus dem \git-Repository, Ordner \datei{tdd} entnommen werden.
Die Zip-Datei muss dafür entpackt werden! Es entstehen die oben 
angeführten Dateien. Bei der Datei \datei{junit5.jar} handelt es sich 
um die Version 1.10.0, die von der Maven-Seite geladen werden kann:

wurden mit dem Terminal.

Bleiben wir aber bei unserem einfachen Lauyout und sehen uns das Testen 
genauer an und zwar die Testklasse. Es geht um eine Funktion, die die
letzten beiden Buchstaben eines Strings vertauschen soll. Das ist zwar 
relativ unsinnig, dafür aber übersichtlich. Wir können sofort identifizieren,
welche Fälle als Tests geprüft werden müssen:

* Ein ganz normaler String wie Hallo
* Strings der Längen 2, 1, 0

Da TDD von einer Minimalismus-Strategie ausgeht, beginnen wir mit dem leeren String.
Werfen wir also einen Blick in die Testdatei:

```bash
package de.alp.kurs;


 import org.junit.jupiter.api.Test;
 import static org.junit.jupiter.api.Assertions.*;


public class StringFlipperTest{
   @Test 
   public void testSwapLastChars(){
      fail("beabsichtigt");
   }
}
```

Der Test soll (nach TDD-Philosophie) Code testen, den es noch nicht gibt und 
deshalb versagen. Wir entfernen die *fail*-Zeile und ersetzen sie durch 
sinnvolleren Inhalt:

```bash
@Test
public void testSwapLastCcharsInEmptyString(){
    StringFlipper stringFlipper = new StringFlipper();
    res = stringFlipper("");
    assertEquals("", res);
    }
```

Die Ausführung des Test scheitert wie gewünscht, da die Funktion noch 
nicht existiert. Ändern wir das:

```bash
public String swapChars(String s) {
        return "";
```

Minimale Problemlösung -- und wirkt wie eine *Veräppelung*.
Dennoch ist das absolut ernst gemeint! Schreiben wir den nächsten Test!

```bash
@Test
public void testSwapLastTwoCharsOnSingleCharString(){
    StringFlipper stringFlipper = new StringFlipper();
    String res = stringFlipper.swapChars("A");
    assertEquals("A",res);
    }
```

Wir können den Test laufen lassen und beobachten sein Scheitern.
Die Funktion muss also verfeinert werden. Beim aktuellen Stand 
wäre das z.B. so möglich:

```bash
public String swapChars(String s) {
    return s;
```

Beide Tests liefern grünen Status! Probieren wir einen String 
mit zwei Buchstaben. 

```bash
 @Test
 public void testSwapLastTwoCharsOnTwoCharString(){
    StringFlipper stringFlipper = new StringFlipper();
    String res = stringFlipper.swapChars("AB");
    assertEquals("BA",res);
    }
```

Der Test scheitert und wir verfeinern die Funktion erneut.
Da es sich um eine neue Situation handelt, trennen wir die Fälle:

```bash
public String swapChars(String s) {
    int length = s.length();
    if (length < 2) {
        return s;
        }
    StringBuilder sb = new StringBuilder();
    sb.append(2);
    sb.append(1);
    return sb.toString();
```


Und wieder *grün*! Nun wollen wir einen vollwertigen 
String probieren!

```bash
@Test
    public void testSwapLastTwoCharsOnNormalString(){
    StringFlipper stringFlipper = new StringFlipper();
    String res = stringFlipper.swapChars("Hallo");
    assertEquals("Halol",res);
    }
```

Wieder zum Verfeinern:

```bash
public String swapChars(String s) {
    int length = s.length();
    if (length < 2) {
        return s;
        }
    StringBuilder sb = new StringBuilder();
    for (int i=0; i < length-2; i++){
        sb.append(s.charAt(i));
    }
    sb.append(s.charAt(length - 1));
    sb.append(s.charAt(length - 2));
    String res = sb.toString();
    return res;
    }
```

Damit haben wir alle Fälle abgehakt, die uns zum Testen eingefallen sind.
Ja -- dieser Prozess ist langatmig und verführt zum Verwenden von 
Abkürzungen. Leider führen diese Abkürzungen am Ende zu schlecht 
programmierter Software! Ein Beispiel ist der Computer der Bosch-eBikes.
Hier ist es möglich, dass negative Fahrzeiten auftreten, wenn man keine 
regelmäßigen Resets der Daten durchführt -- ein Fall, der bei gutem Testing aufgefallen
wäre.

Da Testing so wichtig ist, wird es mittlerweile von allen IDEs unterstützt 
und Build-Tools wie Maven haben es ebenfalls integriert. Für unser eigentliches
Thema CI/CD werden wir aber gleich *Maven* verwenden und uns deshalb vom
manuellen Testen weitgehend verabschieden.





```bash
public class StringFlipperTest{
   @Test 
   public void testSwapLastChars(){
      StringFlipper stringflipper = new StringFlipper();
      assertEquals(1, 1);
   }
}
```

Rerun -- definitiv Red, da es die Klasse noch nicht gibt -- also erstellen wir sie im Ordner \datei{src/java/de/alp/kurs}

```bash
nano ../java/de/alp/kurs/StringFlipper.java 
```

Der Aufruf zum Compilieren muss nun allerdings einen Level nach oben springen!

```bash
javac ../java/de/alp/kurs/StringFlipper.java
```




### Gewohnheiten

* Zuerst das Kriteium formulieren und davon rückwärts arbeiten
* Jeder Test muss zuerst gegen einen definierten Fail laufen.
* Ein Test bekommt nur einen Grund zum Versagen

Damit ein Benutzer einen Text codieren kann, muss zunächst Text
eingegeben werden. Das Programm braucht also eine Eingabe-Methode.



Aus diesem Grund schreibst du jetzt bereits den ersten Test und zwar 
direkt in die \datei{CaesarTest.java}. Die Erklärungen findest du 
unter dem Code :

```java
//imports
import static org.junit.jupiter.api.Assertions.*;
import org.junit.jupiter.api.Test;
import java.lang.reflect.*;

... bisheriger Code ...

@Test
public void inputShouldExist(){  
    Method method = null;       
    try {
        method = Caesar.class.getDeclaredMethod("input"); 
    }
    catch (NoSuchMethodException ignore){
        System.out.println("Methode input fehlt");
    }
    assertNotNull(method, "Methode input fehlt");
}
```

Die Inputs wurden mit WildeCards formuliert, um den Code kürzer 
zu halten. In einer IDE werden die Imports automatisch optimiert!  
In früheren jUnit-Versoinen mussten die Testmethoden sollten mit 
dem Wort *test* beginnen. Das ist mittlerweile wegen `@Test` überflüssig.  

Der Variablentyp *Method* stammt aus dem *reflect*-Package. Die 
Variable *method* wird am Ende verwendet, um den Test als erfolreich 
einstufen zu können. Nimmt sie einen Wert ungleich *null* an, so existiert 
die Methode *input*. Möglicherweise auftretende Fehler werden einfach ignoriert.

Da die Methode noch nicht geschrieben ist, *muss* der Test scheitern. 
Nach TDD-Paradigma musst du den Test ausführen, um ihn scheitern zu sehen.

```bash
mvn test
```

Nun schreibst du den minimalen Code, damit der Test erfolgreich sein wird:

```java
# In der Caesar.java in der Caesar Klasse 
public void input(){}
```

Die Methode macht gar nichts, aber der Test läuft erfolgreich durch.  
Zeit, einen weiteren Test zu schreiben.  

Eine Input-Funktion sollte dazu führen, dass in der Klasse einer Varaiblen ein 
Wert zugewiesen wird. Das soll der nächste Test überprüfen. Da eine derartige 
Variable aber im Regelfall als *private* deklariert ist, kommen wir nicht so einfach 
an sie ran. Ein Getter wäre eine Möglichkeit, besitzt aber im Kontext 
des Programms keinen Sinn.
